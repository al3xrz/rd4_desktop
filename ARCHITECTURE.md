# RD4 Desktop Architecture

Этот документ описывает текущую архитектуру standalone-приложения RD4 Desktop. Его нужно обновлять вместе с изменениями структуры каталогов, слоёв, транзакционных правил, моделей данных, миграций, упаковки и пользовательских потоков.

## Назначение приложения

RD4 Desktop — локальное desktop-приложение для ведения договоров, платежей, актов, строк медицинских услуг, справочника услуг и пользователей. Приложение работает с локальной SQLite-базой, запускает Alembic-миграции при старте и предоставляет интерфейс на PySide2.

## Главный принцип слоёв

Зависимости идут только сверху вниз:

```text
ui -> services -> repositories -> models/core
```

Запрещённые обратные зависимости:

```text
repositories -> ui
models -> services
services -> ui
```

Практически это означает:

- `app/ui/` вызывает сервисы и показывает пользователю результат или ошибку.
- `app/services/` содержит бизнес-операции и управляет транзакциями.
- `app/repositories/` содержит SQLAlchemy-запросы и операции хранения.
- `app/models/` описывает ORM-сущности.
- `app/core/` предоставляет инфраструктуру: настройки, пути, базу данных, миграции, логирование и безопасность.

## Точка входа и запуск

Основная точка входа — `app/main.py`.

Порядок старта:

1. Настраивается логирование.
2. Инициализируется соединение с базой через `app.core.database.init_database`.
3. Запускаются Alembic-миграции через `app.core.migrations.run_migrations`.
4. Создаётся начальный администратор через `app.services.bootstrap.ensure_initial_admin`, если он ещё отсутствует.
5. Импортируется UI-слой и запускается `app.ui.application.run_ui`.

Если PySide2 недоступен, запуск UI не выполняется, а приложение логирует предупреждение.

## Core

Каталог `app/core/` содержит инфраструктурный код.

- `config.py` — настройки приложения и URL базы.
- `paths.py` — вычисление путей к данным, ресурсам и логам.
- `database.py` — SQLAlchemy `Base`, engine, session factory, `session_scope`, SQLite pragmas.
- `migrations.py` — создание Alembic config и запуск миграций.
- `logging.py` — настройка файлового логирования.
- `security.py` — хеширование и проверка паролей.

SQLite pragmas должны оставаться включёнными:

```sql
PRAGMA foreign_keys=ON;
PRAGMA journal_mode=WAL;
PRAGMA synchronous=NORMAL;
```

Расположение базы:

```text
Windows: %APPDATA%\RD4\rd4.db
Linux/macOS: ~/.rd4/rd4.db
Development override: RD4_DATA_DIR=.rd4
```

Окружение разработки и сборки описано в `ENVIRONMENT_SETUP.md`. Целевая версия Python для Windows 7/10 build — Python 3.8.x; `.python-version` фиксирует это значение для `uv` и совместимых менеджеров.

## Models

Каталог `app/models/` содержит SQLAlchemy ORM-модели:

- `User` — пользователи, роли, активность, связи с актами и договорами.
- `Contract` — договор, пациент, представитель, признаки услуг, предоплаты, статус удаления.
- `Payment` — платежи и возвраты по договору, признак проведения и soft-delete.
- `Act` — акт по договору, пользователь-создатель, строки услуг.
- `ActMedService` — строка услуги в акте со снимком кода, названия, единицы, цены, скидки и количества.
- `MedService` — древовидный справочник медицинских услуг и папок.

Модели не должны знать о сервисах или UI.

## Repositories

Каталог `app/repositories/` содержит слой доступа к данным.

Общий базовый класс — `BaseRepository`, специализированные репозитории:

- `ContractRepository`
- `PaymentRepository`
- `ActRepository`
- `ActMedServiceRepository`
- `MedServiceRepository`
- `UserRepository`

Правила:

- Репозитории не открывают транзакции.
- Репозитории не вызывают `commit`.
- Репозитории могут вызывать `flush`, когда нужно получить ID или синхронизировать состояние ORM.
- Репозитории возвращают ORM-объекты или простые результаты запросов.
- Soft-delete используется для бизнес-сущностей, где удаление не должно физически стирать строку.

## Services

Каталог `app/services/` содержит бизнес-логику.

Каждый публичный метод сервиса должен выполнять одну бизнес-операцию в одной service-level транзакции через `session_scope`.

Сервисы:

- `AuthService` — логин, список активных пользователей для login combobox, пользователи, роли, активность, сброс пароля.
- `BootstrapService`/`ensure_initial_admin` — начальное создание администратора.
- `ContractService` — создание, обновление, удаление, сводка одного договора и массовые сводки для реестра.
- `PaymentService` — платежи, возвраты, проведение и отмена проведения.
- `ActService` — акты и строки услуг в актах.
- `MedServiceService` — справочник услуг и папок.
- `DocxService` — генерация и открытие DOCX-документов.

Ошибки домена находятся в `app/services/exceptions.py`. UI должен перехватывать `DomainError` и показывать понятное сообщение пользователю.

## UI

Каталог `app/ui/` содержит PySide2-интерфейс.

Основные части:

- `application.py` — создание QApplication, общий stylesheet, login flow.
- `main_window.py` — главное окно, меню, переключение страниц.
- `login_window.py` — вход пользователя.
- `contracts_page.py` и `contracts_table_model.py` — список договоров.
- `contract_details_page.py` — карточка договора, вкладки платежей и актов.
- `contract_dialog.py` — создание и редактирование договора.
- `payments_panel.py`, `payment_dialog.py`, `payments_table_model.py`, `unpost_payment_dialog.py` — платежи и возвраты.
- `acts_panel.py`, `act_dialog.py`, `act_services_table_model.py`, `act_service_row_dialog.py` — акты и строки услуг.
- `med_services_page.py`, `med_service_dialog.py`, `med_service_picker_dialog.py` — справочник услуг и выбор услуги для акта.
- `users_page.py`, `user_dialog.py`, `password_dialog.py`, `users_table_model.py` — пользователи.
- `icons.py` — локальные и стандартные иконки.
- `qt.py` — единая точка импорта PySide2-классов.

UI-слой не должен выполнять SQL-запросы напрямую. Любые данные для интерфейса приходят через сервисы.

`LoginWindow` показывает активных пользователей в editable combobox, но сохраняет возможность вручную ввести логин.

`ContractsPage` держит список договоров в памяти, фильтрует по периоду, поиску, типу оплаты и статусу баланса, а балансы получает одним агрегированным service/repository-запросом. При открытии карточки договора `MainWindow` оставляет страницу списка живой, а карточку добавляет временной страницей. При возврате из карточки список быстро перечитывается через `reload()`, чтобы балансы после изменений в актах или платежах были актуальны.

`app/ui/icons.py` использует QtAwesome / Font Awesome для основных действий и оставляет fallback на стандартные Qt-иконки и локально отрисованные пиктограммы, если зависимость недоступна.

В главном меню есть разделы `Файл`, `Настройки`, `Отчеты` и `Помощь`. Раздел `Помощь` содержит пункт `О программе`.

Графическая схема взаимодействия модулей и основных UI-сигналов находится в `docs/developer/module_signal_diagram.md`.

Предложения по развитию главной страницы и UX реестра описаны в `MAIN_PAGE_UX_PROPOSAL.md`.

Предложения по развитию вкладки актов описаны в `ACTS_TAB_UX_PROPOSAL.md`.

## Поток создания акта

1. Пользователь открывает договор.
2. На вкладке актов открывается `ActDialog`.
3. При добавлении строки открывается `MedServicePickerDialog`.
4. Пикер показывает дерево справочника, разрешает выбирать только конечные услуги, а не папки.
5. После выбора открывается `ActServiceRowDialog`, где подставляется цена выбранной услуги.
6. Для нового акта строки хранятся в `pending_services` и отображаются в таблице до сохранения.
7. При сохранении `ActService.create_act` создаёт акт и строки услуг в одной транзакции.
8. Для существующего акта добавление, изменение и удаление строк выполняются через методы `ActService`.

## Документы DOCX

`DocxService` отвечает за генерацию документов из шаблонов в `app/templates/`.

Договоры, акты и талоны печатаются через `docxtpl` и получают готовый context из `app/services/docx.py`. Табличные строки актов формируются в самих DOCX-шаблонах через цикл `services`, без отдельного XML-рендерера в сервисе.

Шаблоны должны храниться в репозитории как исходные ресурсы приложения. Сгенерированные документы, временные файлы, логи и локальная база не должны попадать в коммиты.

## Миграции

Alembic находится в `migrations/`.

- `migrations/env.py` — окружение Alembic.
- `migrations/versions/` — версии миграций.
- `alembic.ini` — корневой конфиг.
- `app/core/migrations.py` — runtime-запуск миграций приложения.

После изменения ORM-моделей нужно проверять миграции:

```bash
RD4_DATA_DIR=.rd4 .venv/bin/python -m alembic check
```

Если check показывает новые операции, нужно создать и проверить новую миграцию.

## Тесты

Каталог `tests/` содержит smoke/unit-проверки:

- bootstrap и пользователи;
- database/core;
- модели;
- репозитории;
- сервисы;
- DOCX-сервис.

Базовая проверка перед завершением изменений:

```bash
.venv/bin/python -m pytest tests
RD4_DATA_DIR=.rd4 .venv/bin/python -m alembic check
```

Полный ручной smoke для UI:

```bash
RD4_DATA_DIR=.rd4 .venv/bin/python -m app.main
```

## Packaging

Каталог `packaging/` содержит PyInstaller-конфигурацию:

- `packaging/rd4.spec` — сборка desktop-приложения.
- `packaging/README.md` — заметки по упаковке.

Пошаговая Windows 7 инструкция находится в `WINDOWS7_BUILD_GUIDE.md`.

Проверка упаковки:

```bash
.venv/bin/python -m PyInstaller packaging/rd4.spec --noconfirm
RD4_DATA_DIR=.rd4_bundle_smoke dist/RD4/RD4
```

`build/` и `dist/` не должны коммититься.

Для PyInstaller 4.10 лучше использовать ASCII-путь к исходникам при smoke-сборке на Linux: пути с кириллицей могут ломать поиск PySide2 Qt plugins.

## Soft-delete и безопасность данных

Бизнес-строки не удаляются физически без явного требования.

Soft-delete применяется для:

- договоров;
- платежей;
- актов;
- строк услуг в актах;
- медицинских услуг.

Generated DB files, логи, `.venv/`, `.rd4/`, `build/`, `dist/` не являются исходным кодом приложения и не должны редактироваться вручную или попадать в коммит.

## Совместимость

При изменении зависимостей и синтаксиса нужно учитывать Windows 7/Python 3.8 compatibility. Используемые типы и синтаксис должны оставаться совместимыми с текущей целевой средой проекта.

## Правило поддержки этого документа

При любом изменении структуры приложения нужно обновлять этот файл в том же изменении:

- добавлен, удалён или переименован каталог;
- добавлен новый слой или изменено направление зависимостей;
- добавлен сервис, репозиторий, модель, UI-раздел или значимый пользовательский поток;
- изменены правила транзакций, soft-delete, миграций, базы, путей данных или упаковки;
- изменён порядок запуска приложения.

Если изменение кода не затрагивает архитектуру или структуру, этот файл можно не менять.
