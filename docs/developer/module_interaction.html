RD4 Desktop: структура взаимодействия модулей

Схема показывает, как UI, сервисы, репозитории, модели и инфраструктура связаны между собой. Направление зависимостей остается строгим: ui -> services -> repositories -> models/core.

Слои приложения

Каждый слой отвечает за свой уровень абстракции. UI не делает SQL-запросы, сервисы владеют транзакциями, репозитории только читают и меняют ORM-объекты, модели описывают таблицы, а core дает инфраструктуру.

UI Services Repositories Models Core

app/ui

  • окна и страницы PySide2
  • табличные модели
  • диалоги ввода
  • показ ошибок пользователю

app/services

  • бизнес-операции
  • одна транзакция на операцию
  • валидация правил
  • DomainError для UI

app/repositories

  • SQLAlchemy-запросы
  • create/update/list/get
  • soft-delete
  • flush без commit

app/models

  • ORM-таблицы
  • связи сущностей
  • индексы и ограничения
  • без зависимости от UI

app/core

  • пути и настройки
  • SQLite и сессии
  • Alembic
  • логирование и пароли

Правила чтения схемы

  • Сплошные стрелки показывают прямой вызов или зависимость.
  • Пунктирные стрелки показывают косвенное получение данных через сервисы.
  • UI показывает ошибки только через DomainError, не через SQLAlchemy-исключения.
  • Репозитории не вызывают commit(); транзакцию завершает session_scope().
  • Модели и репозитории ничего не знают о PySide2.
