# Main page UX proposal

Предложение по дизайну и UX главной страницы RD4 Desktop. Цель — сделать стартовый экран быстрым рабочим реестром для оператора, кассира и администратора, без декоративной посадочной страницы.

## Принцип главного экрана

Главная страница должна быть не приветственным экраном, а рабочим реестром договоров.

Пользователь обычно открывает приложение, чтобы:

```text
найти договор -> открыть карточку -> добавить акт или платёж -> вернуться в список
```

Поэтому первый экран должен сразу показывать список договоров, фильтры, быстрые действия и состояние выбранной записи.

## Рекомендуемая структура

```text
Верхнее меню: Файл | Настройки
Статусная строка: пользователь, роль, база, состояние последней операции

Главная область:
1. Заголовок и компактная сводка
2. Панель основных действий
3. Панель фильтров
4. Таблица договоров
5. Нижняя панель выбранного договора
```

## Заголовок и сводка

Верхний блок должен быть компактным:

```text
Договоры
Всего: 3000 | За период: 184 | Долг: 23 | Переплата: 7
```

Сводка полезна, если она считается быстро. Если расчёт дорогой, сначала достаточно:

```text
Всего договоров
Показано по текущему фильтру
```

## Основные действия

Типичный набор кнопок для реестра:

```text
Создать
Открыть
Редактировать
Создать на основе
Удалить
Обновить
```

Рекомендации:

- `Создать` — основное действие, визуально первое.
- `Открыть` — доступно только при выбранной строке.
- `Редактировать`, `Создать на основе`, `Удалить` — доступны только при выбранной строке.
- `Обновить` — полезно после возврата из карточки, импорта или ручной правки базы в тестовой среде.
- Удаление всегда через подтверждение и soft-delete.

Контекстное меню строки должно дублировать основные действия:

```text
Открыть
Редактировать
Создать на основе
Удалить
```

## Фильтры

Обычно в таких проектах нужны:

```text
Поиск
Период
Статус оплаты
Тип оплаты
Категория
Сбросить
```

Минимальный набор для текущего проекта:

```text
Поиск: номер, пациент, телефон, история родов, категория
Период: последние 3 месяца, 6 месяцев, год, все, произвольный
Тип оплаты: все, платно, страховая, не указан
Баланс: все, оплачен, долг, переплата
```

`Сбросить` должен возвращать фильтры к рабочему дефолту:

```text
Период: последние 3 месяца
Поиск: пусто
Тип оплаты: все
Баланс: все
```

## Таблица договоров

Таблица — главный элемент экрана. Рекомендуемые колонки:

```text
Номер
Дата
Пациент
Дата рождения
История родов
Категория
Телефон
Тип оплаты
Баланс
Статус
```

Поведение:

- двойной клик открывает карточку;
- `Enter` открывает выбранный договор;
- сортировка по заголовкам;
- выделение всей строки;
- сохранение выбранной строки при возврате из карточки;
- горизонтальная прокрутка лучше, чем чрезмерное сжатие важных колонок;
- дата, баланс и статус должны быть легко считываемы.

Визуальные статусы:

```text
Оплачен
Долг
Переплата
Без актов
Удалён, если когда-нибудь будет режим показа удалённых
```

Цвета должны быть спокойными:

```text
долг — приглушённый красный
переплата — приглушённый синий или фиолетовый
оплачен — приглушённый зелёный
нет данных — серый
```

Не стоит заливать всю строку ярким цветом. Лучше использовать текст статуса, небольшую метку или цвет только в ячейке баланса.

## Нижняя панель выбранного договора

Сейчас `details_label` показывает краткую информацию. Лучше развить это в компактную панель:

```text
TEST-00042 | Иванова Анна Сергеевна | тел.: +7... | история: ИБ-...
Баланс: -5000.00 | Статус: долг | Тип оплаты: платно
```

Полезные быстрые действия рядом:

```text
Платёж
Возврат
Акт
Печать
```

Если кнопки добавлять рано, можно оставить их в карточке договора, а на главной странице пока показывать только summary.

## Навигация

Текущая модель с `QStackedWidget` подходит:

```text
Договоры -> Карточка договора -> назад к списку
```

Важно сохранить:

- выбранную строку;
- текущий поиск;
- текущий период;
- сортировку;
- позицию прокрутки таблицы.

При возврате из карточки нужно обновлять данные и балансы, но не пересоздавать всё окно.

## Меню

Типичный набор меню для desktop-приложения:

```text
Файл
  Новый договор
  Обновить
  Выход

Настройки
  Справочник услуг
  Пользователи
  Шаблоны документов
  Папка данных
```

`Шаблоны документов` и `Папка данных` особенно полезны для эксплуатации, потому что база и DOCX-шаблоны должны быть доступны администратору системы.

## Статусная строка

В статусной строке полезно показывать:

```text
admin | admin | база: %APPDATA%\RD4\rd4.db
```

После операций:

```text
Договор сохранён
Акт создан
Список обновлён
```

Ошибки лучше показывать модально, но короткий успешный результат — в статусной строке.

## Состояния интерфейса

Нужны понятные состояния:

```text
нет договоров
нет результатов по фильтру
идёт загрузка
ошибка загрузки
нет выбранной строки
```

Для текущего desktop-приложения достаточно текстов без иллюстраций:

```text
Нет договоров за выбранный период
Измените фильтр или создайте новый договор
```

## Горячие клавиши

Полезный минимум:

```text
Ctrl+N — новый договор
Enter — открыть выбранный договор
F5 — обновить список
Ctrl+F — фокус в поиск
Esc — очистить поиск или закрыть диалог
Delete — удалить выбранный договор после подтверждения
```

Клавиши должны дублировать уже видимые действия, а не быть единственным способом выполнить операцию.

## Приоритет внедрения

1. Готово: кнопка `Обновить` и горячая клавиша `F5`.
2. Готово: фильтры `Тип оплаты` и `Баланс`.
3. Готово: колонка `Статус`.
4. Готово частично: выбранный договор сохраняется, список перечитывается без пересоздания страницы.
5. Готово частично: нижняя панель показывает баланс и статус выбранного договора.
6. Готово: пункты меню `Папка данных` и `Шаблоны документов`.
7. Не делаем сейчас: отчёты по долгам и платежам за период.

## Что не стоит делать

- Не делать landing page или крупный приветственный экран.
- Не прятать таблицу за карточками.
- Не использовать яркие декоративные блоки вместо реестра.
- Не перегружать верх страницы большими заголовками.
- Не делать фильтры модальным окном: они должны быть рядом с таблицей.
